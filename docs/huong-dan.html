6. Circuit Breaker — State Machine

File: src/reliability_lab/circuit_breaker.py — Mỗi provider có 1 circuit breaker riêng.

6.1 Ba trạng thái

CLOSED → (fail ≥ threshold) → OPEN → (timeout hết) → HALF_OPEN → (probe OK) → CLOSED
Trạng tháiHành viChuyển sang
CLOSEDCho phép mọi request, đếm failureOPEN khi failure_count ≥ failure_threshold
OPENTừ chối ngay (fail fast), không gọi providerHALF_OPEN sau reset_timeout_seconds
HALF_OPENCho 1 probe request thửCLOSED nếu probe thành công; OPEN nếu probe thất bại

6.2 Các method cần implement

allow_request() → bool

1
CLOSED → luôn return True
2
HALF_OPEN → return True (probe)
3
OPEN → kiểm tra time.monotonic() - opened_at ≥ reset_timeout_seconds
Nếu đủ → transition HALF_OPEN, return True
Nếu chưa → return False

call(fn, *args, **kwargs) → T

1
Kiểm tra allow_request() — nếu False → raise CircuitOpenError
2
Gọi fn(*args, **kwargs)
3
Thành công → record_success(), return kết quả
4
Exception → record_failure(), re-raise

record_success()

  • Reset failure_count = 0, tăng success_count
  • Nếu HALF_OPEN và success_count ≥ success_threshold → transition CLOSED (reason: probe_success)

record_failure() — ⚠️ Phần dễ sai nhất

  • Tăng failure_count, reset success_count = 0
  • if HALF_OPEN → transition OPEN ngay (reason: probe_failure)
  • elif failure_count ≥ failure_threshold → transition OPEN (reason: failure_threshold_reached)
Dùng if/elif, KHÔNG dùng or — hai trường hợp cần reason khác nhau!

6.3 Tham số CircuitBreaker

Tham sốMặc định (config)Ý nghĩa
failure_threshold3Số lỗi liên tiếp trước khi mở circuit
reset_timeout_seconds2Thời gian chờ trước khi thử probe (OPEN → HALF_OPEN)
success_threshold1Số probe thành công để đóng lại circuit

6.4 transition_log

Mỗi lần chuyển trạng thái ghi log: {from, to, reason, ts} — dùng để tính recovery time và đếm circuit_open_count.

7. Cache — ResponseCache & SharedRedisCache

File: src/reliability_lab/cache.py

7.1 similarity(a, b) → float [0..1]

Cosine similarity trên character n-grams + word tokens:

  1. Nếu a == b → return 1.0
  2. Tokenize: tách từ + character 3-grams (vd: "hello" → ["hello","hel","ell","llo"])
  3. Dùng collections.Counter làm vector
  4. Cosine: dot(a,b) / (|a| × |b|) với math.sqrt

7.2 get(query) → (value | None, score)

flowchart TD Q["get(query)"] --> PRIV{"_is_uncacheable?"} PRIV -->|Có| NONE["return None, 0.0"] PRIV -->|Không| EVICT["Xóa entry hết TTL"] EVICT --> SCAN["Tìm entry có similarity cao nhất"] SCAN --> THR{"score ≥ threshold?"} THR -->|Không| MISS["return None, score"] THR -->|Có| FH{"_looks_like_false_hit?"} FH -->|Có| LOG["Ghi false_hit_log, return None"] FH -->|Không| HIT["return value, score"]

7.3 Privacy guardrails — _is_uncacheable()

Regex chặn query chứa: balance, password, credit card, ssn, user.123, account.456...

7.4 False-hit detection — _looks_like_false_hit()

So sánh số 4 chữ số (năm, ID) giữa query và cached key. Nếu khác nhau → từ chối cache hit (tránh trả sai câu trả lời về deadline 2024 vs 2026).

11. Redis Shared Cache

Cho phép nhiều gateway instance chia sẻ cache — in-memory chỉ hoạt động trong 1 process.

Data model Redis

Key:   "{prefix}{query_hash}"     # vd: rl:cache:a1b2c3d4e5f6
Value: Redis Hash { "query": "...", "response": "..." }
TTL:   EXPIRE key ttl_seconds

get() flow

  1. Privacy check → skip nếu uncacheable
  2. Exact match: HGET exact_key response → return (response, 1.0)
  3. SCAN prefix*, HGET query, tính similarity, tìm best ≥ threshold
  4. False-hit check trước khi return

set() flow

  1. Skip nếu uncacheable
  2. HSET key mapping={query, response}
  3. EXPIRE key ttl_seconds
# Kiểm tra Redis
docker compose exec redis redis-cli KEYS "rl:cache:*"
docker compose exec redis redis-cli PING
